# 02. 链表模块 (LinkedList)

本模块包含 **单链表 (Singly Linked List)** 和 **双向链表 (Double Linked List)** 的实现，均采用 **C++17** 标准。

## 1. 单链表 (LinkedList.hpp)
*   **设计**：带虚拟头节点 (`dummyHead`)。
*   **内存管理**：`std::unique_ptr` 管理 `next` 指针。
*   **特点**：插入删除逻辑统一，适合初学者理解智能指针的所有权转移。

## 2. 双向链表 (DoubleLinkedList.hpp)
*   **设计**：**无虚拟头节点** (No Dummy Head)，维护 `head` 和 `tail` 指针。
*   **内存管理**：
    *   `next`: 使用 `std::unique_ptr<Node>` (强引用，拥有所有权)。
    *   `prev`: 使用 `Node*` (裸指针/弱引用)，防止循环引用导致内存泄漏。
*   **核心算法**：
    *   **原地归并排序 (In-place Merge Sort)**：实现了 O(n log n) 的高效排序。
    *   **难点**：在 `unique_ptr` 的限制下，通过 `std::move` 在 `split` 和 `merge` 函数中传递链表的所有权，同时正确维护 `prev` 指针。

### 为什么选择归并排序而不是冒泡排序？

在链表排序中，归并排序优于冒泡排序的核心原因在于**操作的本质**。

#### 冒泡排序的陷阱：交换节点的噩梦
在无虚拟头节点的双链表中，交换两个相邻节点 `A` 和 `B` (`A -> B`) 需要修改 **6 个指针** (`Prev->next`, `B->prev`, `B->next`, `A->prev`, `A->next`, `Next->prev`)。
更糟糕的是**边界条件**：
*   如果 `A` 是头节点，需要更新 `head`。
*   如果 `B` 是尾节点，需要更新 `tail`。
这导致代码充斥着大量的 `if-else`，极易出错且复杂度为 O(n^2)。

#### 归并排序的优雅：重组而非交换
归并排序不试图"修改"两个节点的位置，而是**打散重组**。
1.  **Split (分)**：直接切断链表中间的 `next` 指针。
2.  **Merge (合)**：像整理扑克牌一样，从两堆有序链表中依次取出最小的节点，重新"穿"成一条新链表。
这种方式只需要关心"谁是下一个"，完美避开了复杂的指针交换逻辑，且复杂度为 O(n log n)。

### 4. 非递归归并排序 (Iterative Merge Sort)

为了避免递归带来的栈溢出风险，我们在双链表中实现了**非递归版本**。

#### 核心难点：无虚拟头节点的合并
在不使用 `dummyHead` 的情况下，我们需要一种统一的方式来处理"头节点"和"普通节点"。
我们使用了 **二级指针 (Double Pointer)** 技巧：
*   定义 `std::unique_ptr<Node>* currLink`。
*   它是一个指针，指向**"下一个节点的持有者"** (可能是 `head` 变量本身，也可能是前一个节点的 `next` 成员变量)。
*   通过操作 `*currLink`，我们可以统一代码逻辑，无需为头节点写特殊 `if-else`。

## 3. 运行方法

本目录下的 `Makefile` 支持分别编译两个程序。

```bash
# 1. 编译所有
make

# 2. 运行单链表测试
./LinkedList_main

# 3. 运行双链表测试
./DoubleLinkedList_main

# 清理
make clean
```
